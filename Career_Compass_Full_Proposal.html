Page 2: Executive Summary and Proposal Purpose

Career Compass is proposed as a full-stack, production-oriented digital platform designed to improve how individuals discover, evaluate, and pursue careers that match their personal strengths. The platform solves a common failure point in career planning: users often apply randomly, without structured alignment between interests, personality, capabilities, and labor market demand. This results in inefficient applications, low confidence, and poor role fit.

The proposed solution integrates three user-signal dimensions: MBTI profile, RIASEC profile, and selected skills. These signals are processed through a weighted hybrid scoring engine that calculates role fit and identifies skill gaps. The platform then links users to real openings through job aggregation providers and supports continuity by storing analysis history and path checklists for authenticated users. This turns one-time assessment into an actionable career improvement workflow.

This proposal defines both technical execution and delivery operations. It includes: system architecture, data model, API design, UI/UX behavior, security and privacy approach, risk management, team structure, role assignments across six members, and a 12-week phased implementation plan with measurable outputs. The proposal also documents how technologies are used in detail, how integrations are configured, and how the platform can adapt to custom job APIs through provider abstraction.

Expected outcomes of the 12-week implementation are: a working production-grade MVP, documented setup and operations process, baseline quality assurance coverage, and a demonstrable pipeline from user assessment to opportunity application. Success is measured by functional completeness, reliability of key flows, clarity of user guidance, and maintainability of the codebase for future scale.

Page 3: Problem Statement, Opportunity, and Context

Problem Definition

Current career decision patterns are frequently unstructured. Many learners and job seekers rely on fragmented sources: generic advice articles, informal recommendations, and disconnected job boards. Most tools provide either personality tests without labor-market connection, or job listings without personal-fit guidance. As a result, users do not know which role families to prioritize, which skills to develop first, and where to apply with the highest probability of long-term fit.

Observed Gaps

  • No unified mechanism for combining personality indicators with practical skills.
  • Limited visibility into job requirement gaps at the individual level.
  • Poor continuity: users cannot consistently track progress over time.
  • Weak connection between assessment output and actionable application pathways.
  • Lack of transparent scoring logic in many recommendation systems.

Opportunity

Career Compass addresses these gaps by introducing a structured and transparent model that connects profile analysis to real jobs. It does not attempt to replace human coaching; instead, it provides an evidence-based decision support layer for users to move faster and more confidently. The opportunity includes educational institutions, bootcamp learners, graduates, early-career professionals, and career-switchers who need practical and personalized direction.

Value Proposition

The platform delivers practical value in three ways: first, it narrows career options to high-fit targets; second, it transforms requirements into a clear checklist and upskilling path; third, it connects users to live opportunities. The combined effect is better application quality, reduced decision fatigue, and improved readiness for role transitions.

Page 6: System Architecture (High-Level)

Architecture Pattern

The solution follows a layered, modular architecture with clear boundaries:

  • Presentation Layer: React frontend handling UI, routing, and user interactions.
  • API Layer: Express routes/controllers handling request orchestration and validation.
  • Domain/Service Layer: business logic (scoring, sync, checklist workflows).
  • Data Access Layer: repositories with SQL operations via PostgreSQL pool.
  • Persistence Layer: PostgreSQL tables with constraints, indexes, and migrations.

Request Lifecycle Example

A career analysis request starts at frontend selection inputs and moves through API validation, scoring services, lookup enrichment, optional user history recording, then returns ranked results. This keeps frontend thin while preserving backend consistency and testability.

Operational Flow for Jobs

Job sync is triggered via API. Provider adapter fetches raw jobs, normalization transforms to unified schema, career ranking maps jobs to known paths, and repository upserts records into `jobs`. Frontend then reads normalized jobs through a single endpoint, independent of source provider.

Scalability and Extensibility Notes

  • Provider abstraction allows adding new APIs without frontend changes.
  • Repository pattern centralizes SQL and supports gradual optimization.
  • Stateless API routes support horizontal scaling with external sessionless auth.
  • Migrations maintain schema evolution discipline for team collaboration.

Page 7: Frontend Technical Design (Used and How)

Core Technologies Used

  • React with Vite build tooling.
  • Hash-based client routing (custom hook).
  • Firebase Auth SDK for identity and session state.
  • Custom CSS variable system for theme and visual consistency.

How It Is Implemented

The frontend is organized by responsibility: `pages` for route-level screens, `components` for reusable UI, `services` for API calls, `context` for auth state, and `hooks` for routing/SEO behavior. This structure reduces coupling and supports iterative feature growth.

Navigation relies on hash routes (`#/home`, `#/analysis`, etc.) to avoid server-side routing dependencies during local and static deployments. The custom route hook normalizes valid routes and redirects invalid paths to home.

Theme management is state-driven. On app initialization, theme is loaded from local storage or user system preference. Toggling updates `document.documentElement.dataset.theme`, enabling CSS variable swapping for light/dark mode without component rewrites.

Auth UX Flow

  • Dedicated Login page presents Google and email/password options.
  • Firebase state listener updates app-level auth context.
  • Token is attached to protected API requests through service client options.
  • Logout clears session and user context, then guarded pages show proper state.

UI/UX Principles Applied

  • Glassmorphism cards with clear hierarchy and readable contrast.
  • Responsive behavior for mobile and desktop layouts.
  • Safe external links (`noopener noreferrer`) for test links/job links.
  • Progressive interaction: analysis output, then requirement path tracking.

Error Handling and Stability

Controllers throw typed HTTP errors for invalid payloads or authorization failures. Global error middleware ensures consistent JSON responses and avoids leaking stack traces to clients in standard operation. Job provider failures use controlled fallback behavior where configured to reduce user disruption.

Authentication Verification Path

Optional Firebase Admin verification validates bearer tokens when credentials are configured. If not configured, mock fallback can be retained for local development. Protected routes use middleware that requires resolved user identity before allowing tracking/history operations.

Page 9: Database Design and Data Governance

Schema Overview

The schema models three major domains: (1) career intelligence (careers, skills, certifications, mappings), (2) market data (jobs, job sources, world stats), and (3) user progression (analysis history, checklist items).

Key Modeling Choices

  • Many-to-many tables capture career-to-skill, career-to-mbti, and career-to-riasec weights.
  • Normalized requirement records support transparent recommendation explanations.
  • JSONB fields preserve provider payloads for traceability and future enrichment.
  • Unique constraints prevent duplicate job ingestion from repeated sync runs.

Data Quality Controls

  • Check constraints enforce valid enumerations (experience level, checklist item type).
  • Indexed foreign keys support fast joins on recommendation and path queries.
  • Seed scripts use upsert patterns for idempotent repeatability.
  • Migrations are tracked in `schema_migrations` to preserve sequence integrity.

Seed Strategy

The current seed includes more than 100 skills and at least 120 baseline jobs to ensure functional demonstration even during upstream API instability. This prevents empty states during demonstrations and initial tests while preserving compatibility with live sync providers.

Page 10: Recommendation Engine and Decision Logic

Hybrid Weighted Scoring Formula

  • Skill match coverage: 50%
  • RIASEC alignment: 30%
  • MBTI alignment: 20%

This weighting was selected to balance practical employability (skills) with role-style compatibility (personality indicators). It intentionally prioritizes tangible capability while preserving behavioral fit factors.

Scoring Steps

  1. Normalize user-selected inputs (skills, MBTI, RIASEC).
  2. Load active career profiles with weighted mappings.
  3. Compute partial coverage across each scoring dimension.
  4. Aggregate weighted total and bound to percentage scale.
  5. Sort and return top recommendations with breakdown details.

Skill Gap Logic

For each recommended career, required skills not selected by user are flagged as skill gaps. Gaps are sorted by importance, enabling users to prioritize upskilling in high-impact areas first.

Explainability and Trust

Each recommendation includes breakdown data (skill, RIASEC, MBTI contribution), matched signals, and required certifications. This transparency avoids black-box recommendations and improves user trust and self-directed action planning.

Page 11: External Integrations (Jobs APIs, Firebase, and Custom Provider)

Jobs Integration Strategy

Career Compass supports multiple job providers under a single normalization pipeline. The provider is controlled by environment configuration:

  • `remotive`: public remote jobs API
  • `adzuna`: requires `app_id` and `app_key`
  • `custom`: user-defined endpoint and field path mapping

Adzuna Configuration (Current Credential Model)

Requests include credentials in query parameters (`app_id`, `app_key`) per Adzuna API style. Country and page are configurable for market scope and pagination control.

Custom Provider Configuration

For custom APIs, the system supports method control (`GET`/`POST`), custom headers JSON (for tokens), results path extraction (e.g. `data.items`), and search/limit parameter naming. This allows integration with proprietary job backends without changing core business logic.

Firebase Integration

Frontend uses Firebase Authentication for Google and email/password login. Backend optionally verifies Firebase ID tokens using Firebase Admin credentials for protected user tracking operations.

Failure Handling

If provider fetch fails, controlled fallback jobs can be injected (for default providers) to avoid total user-blocking errors. Failures are returned as structured API responses for diagnostics while maintaining platform continuity.

Page 12: Security, Privacy, Compliance, and Reliability

Security Controls

  • Input validation at API boundary (`zod`).
  • Helmet-based security headers.
  • CORS restricted by configured origin.
  • Authenticated route protection for history/checklist endpoints.
  • Sensitive credentials moved to environment variables.

Privacy and Data Handling

User tracking data is limited to platform-relevant activity: selected inputs, top recommendations, and checklist completion status. Personally identifiable information scope is minimized to authenticated account identity and required metadata for feature operation.

Operational Reliability

  • Health endpoint enables runtime checks.
  • Migration/seed scripts provide deterministic environment setup.
  • Seed baseline prevents empty datasets during API outages.
  • Single command startup script reduces local run-time errors.

Compliance Considerations

In future phases, policy upgrades should include explicit data retention policy, account deletion workflow, audit logging for access-sensitive actions, and legal review for multi-region privacy obligations where applicable.

Deployment Plan (MVP)

  2. Configure production env values for Firebase and jobs providers.
  3. Run migrations and seed scripts in target environment.
  4. Deploy backend and frontend with proper CORS and base URL configuration.
  5. Execute smoke tests and publish release notes.

Post-Deployment Monitoring

Track API error rates, sync success rates, auth failures, and user progression usage. This telemetry informs tuning of scoring assumptions and next-phase product decisions.
